## ВВЕДЕНИЕ

Современные информационные системы в сфере экономики и управления всё чаще строятся на микросервисной архитектуре с REST API в качестве точки интеграции. Платформа Node.js благодаря асинхронной модели ввода-вывода и единому стеку JavaScript/TypeScript широко применяется для построения высоконагруженных веб-сервисов. При этом выбор серверного фреймворка и режима исполнения приложения (один процесс, кластер процессов или вынос вычислений в worker threads) существенно влияет на пропускную способность, задержку ответа и устойчивость системы под нагрузкой.

**Актуальность** исследования обусловлена необходимостью обоснованного выбора технологического стека при проектировании корпоративных API: ошибочное решение может привести к росту затрат на инфраструктуру, деградации пользовательского опыта и нарушению SLA.

**Цель практики** — исследовать и сравнить производительность популярных Node.js-фреймворков (Express, Fastify, NestJS) при обработке высокой нагрузки в однопоточном режиме (Event Loop) и в многопроцессной/многопоточной конфигурации, выявить оптимальные сценарии использования каждого фреймворка в зависимости от характера нагрузки (I/O-bound vs CPU-bound).

1. Изучить теоретические основы архитектуры Node.js, модулей `cluster` и `worker_threads`, а также особенностей Express, Fastify и NestJS.

## 1 ТЕОРЕТИЧЕСКИЕ ОСНОВЫ И ПОСТАНОВКА ИССЛЕДОВАНИЯ

### 1.1 Архитектура Node.js и модели параллелизма

Node.js — среда выполнения JavaScript на движке V8, построенная вокруг **неблокирующей модели ввода-вывода** и **единственного потока Event Loop** в каждом процессе. Синхронные CPU-интенсивные операции блокируют цикл событий и увеличивают задержку для всех клиентов; I/O-операции делегируются libuv и не блокируют поток.

Для масштабирования на многоядерных CPU применяют:

- **модуль `cluster`** — master-проcess порождает несколько worker-процессов, каждый со своим Event Loop; входящие соединения распределяются ОС (SO_REUSEPORT / round-robin);
- **модуль `worker_threads`** — лёгкие потоки внутри одного процесса для CPU-bound задач без полного дублирования памяти процесса.

Типичная классификация нагрузки:

| Тип | Пример | Риск для single process | Стратегия |
|---|---|---|---|
| I/O-bound | запрос к БД, внешнему API | низкий при async/await | single или cluster для RPS |
| CPU-bound | хэширование, обработка массива | блокировка Event Loop | cluster или worker_threads |
| Memory-bound | парсинг большого JSON | GC-паузы, рост RAM | ограничение payload, streaming |

### 1.2 Обзор сравниваемых фреймворков

**Express** (4.x) — де-факто стандарт middleware-архитектуры для Node.js; минималистичный роутинг, большая экосистема, умеренный overhead.

**Fastify** (4.x) — фреймворк с акцентом на производительность: схемы JSON Schema, низкий overhead маршрутизации, встроенная сериализация.

**NestJS** (10.x) — enterprise-фреймворк на TypeScript с dependency injection, декораторами и модульной архитектурой; дополнительный overhead ради структурированности и масштабируемости кодовой базы.

В рамках практики все три сервера реализуют **идентичный контракт API** и общую бизнес-логику (`shared/`), что обеспечивает корректность сравнения.

### 1.3 Метрики и методика нагрузочного тестирования

**Метрики сравнения** (согласно индивидуальному заданию, вариант № 14):

- пропускная способность — requests per second (RPS);
- время отклика — среднее, медиана, перцентиль p95;
- нагрузка CPU и потребление RAM (фиксировались через pidusage; на Windows в cluster-режиме значения RAM оказались нулевыми из-за ограничений сбора — см. раздел 3).

**Инструмент:** k6 v2.0.0 — современный load testing tool с сценариями на JavaScript, поддержкой constant-arrival-rate и ramping VUs.

**Типовые кейсы** (6 сценариев):

1. GET `/api/static` — статический JSON;
2. POST `/api/transform` — приём и трансформация тяжёлого JSON (~10–15 МБ);
3. GET `/api/io-simulate?delay=300` — имитация async I/O 300 мс;
4. POST `/api/hash` — SHA-256 (CPU);
5. POST `/api/compute` — обработка массива до 10⁷ элементов (CPU);
6. стресс-тест `/api/counter` — конкурентное чтение/запись (500 VUs, ramp-up).

## 2 РАЗРАБОТКА ЭКСПЕРИМЕНТАЛЬНОГО СТЕНДА

### 2.1 Архитектура тестовых HTTP-сервисов

Проект организован как npm-workspace монорепозиторий:

- `shared/` — общая бизнес-логика и `cpu-worker.js`;
- `servers/express/` — порт 3001;
- `servers/fastify/` — порт 3002;
- `servers/nestjs/` — порт 3003 (TypeScript, сборка в `dist/`);
- `load-tests/` — k6-сценарии;
- `scripts/run-benchmarks.js` — оркестратор 78 прогонов (профиль standard).

Единый набор эндпоинтов: `/health`, `/api/static`, `/api/transform`, `/api/io-simulate`, `/api/hash`, `/api/compute`, `/api/counter` (GET/POST increment/reset).

### 2.2 Реализация режимов single, cluster и worker_threads

**Single** — один процесс Node.js, один Event Loop (`node app.js`).

**Cluster** — master fork-ит 4 worker-проcessa; каждый слушает тот же порт через модуль `cluster`.

**Workers** — переменная окружения `USE_WORKER_THREADS=true`; CPU-операции hash и compute выполняются в пуле worker threads, не блокируя основной Event Loop.

Для NestJS кластеризация реализована обёрткой над `cluster` при bootstrap; worker_threads подключаются через общий `shared/cpu-worker.js`.

### 2.3 Автоматизация прогонов и сбор метрик

Скрипт `run-benchmarks.js`:

1. запускает сервер в нужном режиме;
2. ожидает `/health`;
3. выполняет k6-сценарий с заданным target RPS;
4. парсит summary JSON (формат k6 v2);
5. записывает результат в `results/benchmark-*.json`;
6. генерирует отчёт `results/reports/report-*.md` и CSV.

Полный прогон **14.06.2026**: 78/78 тестов, 0 ошибок, источник — `benchmark-1781438064444.json`.

## 3 ПРОВЕДЕНИЕ НАГРУЗОЧНЫХ ИСПЫТАНИЙ И АНАЛИЗ РЕЗУЛЬТАТОВ

### 3.1 Результаты I/O-bound сценариев

#### 3.1.1 Статический GET (static)

При target RPS 1000 все три фреймворка достигли ~1000 req/s с p95 ≤ 0,5 мс. Fastify (cluster) показал 1000,01 req/s и p95 = 0,00 мс; Express и NestJS — сопоставимые значения. **Вывод:** на лёгких GET-запросах overhead фреймворка минимален; узким местом является сеть и k6, а не сервер.

#### 3.1.2 Имитация I/O (io-simulate, delay=300 мс)

При target 500 RPS actual RPS ≈ 492 для всех фреймворков; p95 ≈ 301–312 мс — определяется искусственной задержкой, а не фреймворком. **Вывод:** при I/O-bound нагрузке выбор фреймворка не влияет на latency, если Event Loop не блокируется.

### 3.2 Результаты CPU-bound сценариев

#### 3.2.1 Хэширование SHA-256 (hash)

При 100 RPS в режимах **single/cluster** p95 = 1,3–2,5 мс (Express single — 2,50 мс; Fastify cluster — 1,43 мс).

#### 3.2.2 Числовые вычисления (compute, массив 10⁷)

При 30 RPS single/cluster: p95 = 3–4 мс. При workers: p95 ≈ 24–25 мс — аналогичный эффект overhead. **Cluster** при 10 RPS compute даёт умеренное улучшение за счёт распределения по ядрам.

**Ключевые выводы:**

1. **Fastify** — лидер по RPS и p95 на concurrency и static; рекомендуется для high-RPS API без тяжёлой enterprise-обвязки.
2. **Express** — стабилен, но уступает Fastify в ~3,5 раза по RPS на стресс-тесте; оправдан для legacy и прототипов.
3. **NestJS** — компромисс между архитектурой и производительностью; cluster даёт ~2,7× прирост RPS относительно single.
4. **Cluster** эффективен для I/O-light и concurrency; для io-simulate прирост минимален.
5. **Worker threads** в данном эксперimente **ухудшили** p95 на лёгких CPU-задачах.
6. In-memory counter в cluster **не гарантирует** глобальную консистентность между процессами — для production нужен Redis/БД.

**Ограничения эксперimenta:**

- I/O имитирован `setTimeout`, не реальной СУБД;
- RAM/CPU метрики pidusage = 0 на Windows/cluster;
- тесты на localhost — без сетевой латентности WAN;
- один прогон на конфигурацию — для научной публикации желательны повторы и доверительные интервалы.

**Сводка рекомендаций по выбору стека:**

| Сценарий | Рекомендация |
|---|---|
| Максимальный RPS, лёгкие GET | Fastify + cluster |
| Enterprise, DI, TypeScript | NestJS + cluster |
| Legacy, простота | Express + cluster при росте нагрузки |
| Длительные CPU-задачи | cluster или workers (после профилирования порога) |
| Консистентное состояние | внешнее хранилище, не in-memory cluster |

## ЗАКЛЮЧЕНИЕ

В ходе научно-исследовательской работы выполнено сравнительное исследование производительности фреймворков Express, Fastify и NestJS на платформе Node.js v22 в режимах single process, cluster (4 workers) и worker_threads.

Разработан единый экспериментальный стенд с шестью типовыми сценариями нагрузки, проведено 78 автоматизированных прогонов k6 без ошибок. Установлено, что **Fastify** обеспечивает наивысшую пропускную способность (до 53 584 req/s в cluster-режиме на тесте конкурентности) и наименьшую задержку p95 на I/O-light сценариях. **NestJS** занимает промежуточное положение, предлагая архитектурные преимущества ценой 10–30 % overhead. **Express** остаётся работоспособным, но уступает по raw performance.

Подтверждено, что **характер нагрузки** определяет оптимальный режим: для I/O-bound задач фреймворк вторичен; для CPU-bound коротких операций worker_threads контрпродуктивны; cluster масштабирует throughput на многоядерных системах.

**Направления дальнейших исследований:** повторение эксперimenta на Linux-сервере; сбор RAM/CPU через clinic.js или Prometheus; тестирование с реальной PostgreSQL; сравнение с Bun/Deno; исследование порога длительности CPU-задачи, при котором worker_threads становятся выгодными.

Практика позволила сформировать компетенции ОПК-1.3, ОПК-2.2, ОПК-2.3, ОПК-3.3 и ОПК-4.1 в части экспериментального исследования, применения ИТ-инструментов и оформления научно-технической документации.
